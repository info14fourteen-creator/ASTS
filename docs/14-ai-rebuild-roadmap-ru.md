# План восстановления АСТС с AI

Дата: 2026-06-11.

Цель: воссоздать проверенную B2G/АСТС-логику из блок-схемы, но заменить максимальную часть ручной рутины и работы фрилансеров AI-модулями.

## Главный принцип

Мы не делаем “новый AI-парсер тендеров”. Мы восстанавливаем старую рабочую операционную систему тендерной сделки:

> источники закупок -> база процедур и документов -> AI-разбор -> workflow ролей -> поставщики -> КП -> экономика -> решение -> подача -> исполнение -> закрытие.

AI должен заменить рутину, но не уничтожить проверенную логику:

- статусы остаются;
- роли остаются;
- подтверждения остаются;
- дедлайны остаются;
- причины отказов остаются;
- top-3 поставщиков остаются;
- расчет рентабельности остается;
- audit log обязателен.

## Что переносим из блок-схемы без потери

### Две воронки сделки

В новой ASTS нельзя держать старую карту одной длинной стадией. Должны быть две отдельные воронки:

- `TenderParticipationFunnel` - до победы в процедуре;
- `ContractExecutionFunnel` - после победы, исполнение.

Первая воронка: участие в процедуре.

1. Формирование начальных условий для подбора процедур.
2. Выгружена из базы.
3. Оценка возможности участия.
4. Генерация запроса КП.
5. Ожидание ответа поставщиков.
6. Анализ полученного предложения.
7. Внесение цен.
8. Оферта.
9. Акцептирование.
10. Расчет рентабельности.
11. Подача документов.

Финалы первой воронки:

- победили в процедуре;
- проиграли в конкурентной борьбе;
- отказались до подачи;
- вышел срок;
- нерентабельная сделка;
- отменена заказчиком;
- ручное управление.

Вторая воронка: исполнение после победы.

1. Договор, спецификация, счет.
2. Счет, оплата.
3. Закупка у поставщика.
4. Исполнение.
5. Закрывающие документы.
6. Репутационные документы.
7. Финальный расчет.

MVP не обязан реализовать исполнение, но база данных и статусы должны быть спроектированы так, чтобы победившая процедура переходила во вторую воронку без переделки.

### Неуспешные финалы и причины

Сохраняем как справочники:

- отсутствие поставщика/подрядчика;
- вышел срок;
- нерентабельная сделка;
- проиграли в конкурентной борьбе;
- отменена по решению заказчика;
- отказались по решению руководства;
- прочие причины;
- не найден исполнитель;
- все исполнители отказались;
- все исполнители заняты;
- не рассчитались;
- не приняли продукцию/результаты работы;
- штрафные санкции;
- возврат денежных средств;
- частичная оплата.

### Роли

Роли старой системы:

- менеджер;
- специалист по работе с государственным сегментом;
- специалист юридической службы;
- финансовый контролер;
- бухгалтер;
- специалист службы безопасности;
- специалист отдела закупок;
- специалист отдела продаж;
- специалист отдела делопроизводства;
- специалист отдела логистики;
- фрилансер/категорийный менеджер;
- поставщик.

В новой системе AI заменяет часть задач фрилансера и часть первичного анализа, но роли остаются как точки контроля.

## Где AI заменяет фрилансеров и рутину

### Вместо фрилансера по товарным позициям

AI делает:

- читает архив документации;
- извлекает товарные позиции;
- определяет количество, единицу, характеристики;
- предлагает ОКПД2/категорию;
- отмечает спорные/неуверенные позиции;
- создает таблицу позиций.

Человек делает:

- проверяет только low-confidence позиции;
- подтверждает аналоги;
- блокирует грубые ошибки.

### Вместо ручного первичного анализа

AI делает:

- краткую карточку процедуры;
- требования к участнику;
- список обязательных документов;
- сроки;
- обеспечение заявки/контракта;
- риски;
- вопросы для специалиста;
- рекомендацию: участвовать, отказ, ручная проверка.

Человек делает:

- финальное решение;
- юридическую/коммерческую ответственность;
- ручной режим, если AI или экономика показывают риск.

### Вместо ручного запроса КП

AI делает:

- генерирует запрос КП;
- формирует email;
- формирует PDF/HTML;
- готовит публичную страницу поставщика;
- подсказывает, какие характеристики нельзя менять;
- объясняет поставщику, как предлагать аналоги.

Человек делает:

- подтверждает список поставщиков;
- подтверждает отправку;
- проверяет спорные аналоги.

### Вместо ручного сравнения КП

AI делает:

- читает PDF/XLSX/скан КП;
- нормализует цены;
- сопоставляет позиции;
- выделяет аналоги;
- отмечает неполные КП;
- готовит top-3.

Система делает:

- считает min/mid/max сценарии;
- проверяет НМЦК;
- считает рентабельность;
- создает задачи по недостающим затратам.

## Источники данных

### Правило источников

Продукт строим только на первоисточниках:

- официальные государственные источники;
- официальные API/выгрузки электронных торговых площадок;
- официальные реестры и открытые данные;
- документы, скачанные из первичного источника.

Агрегаторы не используем как источник истины. Они допустимы только для исторического импорта старых файлов, ручной сверки полноты или миграции.

Подробное правило: `docs/15-primary-source-policy-ru.md`.

### 1. ЕИС / zakupki.gov.ru

Официальные точки входа, которые нужно поддержать:

- поиск закупок 44-ФЗ/223-ФЗ;
- карточки закупок;
- документы закупки;
- реестр контрактов 44-ФЗ;
- реестр договоров 223-ФЗ;
- реестр организаций;
- открытые данные ЕИС.

Нужно сделать отдельный модуль `data_sources/eis`.

На старте не закладываться на один способ доступа. Реализовать адаптеры:

- `EisSearchScraper` - аккуратный сбор из публичных страниц поиска;
- `EisXmlNoticeClient` - получение XML/print-form по номеру извещения, где доступно;
- `EisOpenDataImporter` - загрузка открытых данных, если доступен нужный набор;
- `EisDocumentDownloader` - скачивание файлов документации;
- `EisContractRegistryImporter` - история контрактов/победителей;
- `EisOrganizationImporter` - заказчики и организации.

Важно: у ЕИС исторически менялись способы доступа, поэтому модуль источника должен быть заменяемым. Все данные сохраняем у себя в нормализованном виде.

### 2. ФНС / nalog.gov.ru

Официальные источники для проверки заказчиков, поставщиков и контрагентов:

- ЕГРЮЛ/ЕГРИП через официальный режим интеграции;
- официальные открытые данные ФНС;
- сервисы ФНС по контрагентам, если условия использования позволяют интеграцию;
- ФИАС/адресные данные при необходимости нормализации адресов.

Нужно сделать отдельный модуль `data_sources/fns`.

### 3. Официальные электронные торговые площадки

Подключаем только официальные API, выгрузки, кабинеты или разрешенные способы интеграции самих площадок.

Кандидаты:

- Сбербанк-АСТ;
- РТС-тендер;
- Росэлторг;
- ТЭК-Торг;
- ЭТП ГПБ;
- АГЗ РТ;
- АСТ ГОЗ;
- другие площадки из официального перечня ЕИС.

Нужно сделать модуль `data_sources/etp`.

### 4. Исторические импорты Excel/PDF

У нас уже есть старые XLSX-выгрузки Tenderland. Это быстрый путь для MVP:

- импорт XLSX;
- нормализация полей;
- привязка к источнику;
- скачивание/прикрепление документов вручную или по ссылке.

Важно: такие записи получают тип источника `aggregator_import` или `manual_import`. Они не должны подменять данные из ЕИС, ФНС или ЭТП.

### 5. Коммерческие и резервные источники

Полезные варианты на будущее:

- Контур.Закупки / Контур.Фокус;
- Seldon;
- Tenderplan;
- Tendee;
- DaData;
- ФНС ЕГРЮЛ/ЕГРИП;
- Ofdata/NewDB/Damia или другие API по юрлицам и контрактам;
- собственная база поставщиков из старых сделок.

Такие источники не должны быть hard dependency MVP и не должны становиться источником истины. Делаем интерфейсы `TenderSourceProvider`, `OfficialSourceProvider` и `CompanyEnrichmentProvider`.

### CRM и учетные интеграции

Нужен общий `CRM hub`, чтобы не писать бизнес-логику отдельно под каждую систему.

Первая очередь:

- Bitrix24;
- amoCRM;
- 1C.

Вторая очередь:

- Мегаплан;
- RetailCRM;
- Planfix;
- СберCRM;
- C2CRM;
- РосБизнесСофт CRM;
- BPMSoft CRM.

Уровни интеграции:

1. Export/link mode: передаем ссылку на карточку ASTS и создаем задачи.
2. One-way sync: ASTS отправляет сделки, компании, контакты, задачи, комментарии и статусы.
3. Two-way sync: двусторонние статусы, ответственные, задачи, комментарии, файлы.
4. CRM app/widget: карточка ASTS внутри CRM.

Каждый CRM-коннектор должен проходить отдельную проверку официального API, лимитов, OAuth/токенов, webhooks и marketplace-правил.

## Этапы работы

### Phase 0: Зафиксировать логику

Цель: не потерять старую B2G-логику.

Сделать:

- принять PR с анализом RTB/PDF;
- создать issues из этого плана;
- описать workflow statuses;
- описать role matrix;
- описать escalation policies;
- описать cost formulas как настройки, а не хардкод.

Готово, когда:

- есть docs по workflow;
- есть backlog issues;
- команда понимает, что входит в MVP, а что позже.

### Phase 1: База продукта и API

Цель: создать ядро, куда можно складывать тендеры, документы, позиции и задачи.

Сделать:

- SQLAlchemy/Alembic;
- модели `Tender`, `TenderDocument`, `TenderPosition`;
- модели `Task`, `TaskTemplate`, `EscalationPolicy`;
- модели `Organization`, `User`, `RoleAssignment`;
- seed справочников ролей и статусов;
- health/API docs;
- тестовые fixtures.

Готово, когда:

- API локально запускается;
- можно создать тендер;
- можно добавить позиции;
- можно назначить задачи;
- можно видеть статусы.

### Phase 2: Источники данных

Цель: начать собирать закупки и документы у себя.

Сделать:

- модуль `data_sources/eis`;
- модуль `data_sources/fns`;
- модуль `data_sources/etp`;
- spike по ЕИС: поиск, карточка, XML/JSON/print-form, документы;
- downloader документов;
- official source raw storage;
- нормализация карточки закупки;
- ручное создание тендера по URL;
- миграционный XLSX/PDF importer для старых файлов без статуса источника истины.

Готово, когда:

- можно получить или сохранить процедуру из официального источника;
- можно сохранить карточку закупки;
- можно скачать или прикрепить документы;
- все raw-файлы лежат в S3/MinIO;
- у каждой записи есть `source_system`, `external_id`, `source_url`, `raw_payload_hash`.

### Phase 3: AI вместо фрилансера

Цель: AI извлекает позиции и требования.

Сделать:

- document parser PDF/DOCX/XLSX/ZIP;
- chunking;
- AI extraction schemas;
- extraction job;
- confidence scores;
- source references;
- UI проверки позиций;
- temporary review URL.

Готово, когда:

- загружаем документы;
- AI создает таблицу позиций;
- AI создает список требований;
- человек видит спорные места и подтверждает.

### Phase 4: Поставщики и КП

Цель: восстановить закупочную сторону старой логики.

Готово, когда:

## Решение по MVP

MVP должен покрыть путь:

> импорт/сбор тендера -> документы -> AI-разбор -> позиции -> поставщики -> КП -> top-3 -> рентабельность -> решение.

Не включаем в MVP:

- полную подачу документов на площадки;
- договор/спецификацию/счет;
- исполнение;
- закрывающие документы;
- репутационные документы;
- финальный расчет после исполнения;
- полноценный marketplace фрилансеров.

Но модель данных должна предусматривать эти будущие этапы.
