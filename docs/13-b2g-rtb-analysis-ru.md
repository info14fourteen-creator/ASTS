В PDF явно перечислены исходы:

- отсутствие поставщика/подрядчика;
- вышел срок;
- нерентабельная сделка;
- проиграли в конкурентной борьбе;

- отказались по решению руководства;

- штрафные санкции;
- возврат денежных средств;

Есть также статусы/сценарии запросов разъяснений:

- влияние на сделку со стороны конкурентов;
- влияние на сделку со стороны заказчика;
- влияние на сделку с нашей стороны.

### Сущности Bitrix/CRM, которые были в старой логике

PDF явно опирается на Bitrix-сущности:

- сделка;
- компания;
- контакт;
- товар;
- товарная позиция;
- списки.

Назначение ролей строилось через компанию-партнера и контакт:

1. Для сущности `Сделка` выбирается поле `Компания партнер`.
2. Для сущности `Компания` проверяются контакты.
3. Для сущности `Контакт` выбирается поле `Роль для работы`.
4. Контакт с ролью `Менеджер` назначается на сделку.
5. Если контакт не найден, назначается дефолтный ответственный.

Для нового ASTS это означает: импорт из CRM должен уметь подтягивать компанию, контакты и роли, но внутри продукта лучше хранить свои `Organization`, `User`, `RoleAssignment`.

### Автоматические проверки времени

Есть ежедневная проверка:

- запуск в 04:00 каждый день;
- проверка воронки `B2G`;
- если `Дата и время окончания подачи заявок` раньше текущего времени сервера + 10 часов, сделка переносится в статус `Вышел срок`.

Есть формула дедлайна в часах:

```text
24 * datediff(now, submission_deadline, days) + datediff(now, submission_deadline, hours)
```

Также встречается правило:

- если сделка в стадии `Внесена`;
- и до окончания срока подачи заявок меньше 10 часов;
- назначить неудачный финал `Вышел срок`;
- отправить супер-администратору список удаленных/закрытых сделок.

### Эскалация задач

В PDF есть общий паттерн:

- задача назначается рядовому сотруднику;
- если он игнорирует, ответственный меняется на менеджера;
- цикл повторяется до лимита;
- если менеджер тоже не реагирует, процедура уходит в отказ по решению руководства.

Для отдельных ролей правила разные:

- специалист по гос. сегменту просрочил - задача менеджеру, специалист переназначается;
- специалист отдела продаж просрочил - задача менеджеру, специалист переназначается;
- фрилансер просрочил - задание отдается другому фрилансеру;
- закупщик просрочил проверку аналогов - местами автоодобрение, местами ничего не происходит;
- логист просрочил - система ставит 4% от стоимости товаров;
- менеджер просрочил бонус/командировку - не учитываем;
- гос. специалист просрочил обеспечение/удержания - передаем менеджеру или не учитываем в зависимости от задачи.

Это надо реализовать как `TaskTemplate`/`EscalationPolicy`, а не хардкодить в одном статусе.

### Одобрение найденных процедур

Есть отдельный daily-flow:

- система сообщает, сколько новых процедур найдено за день и на какую сумму;
- пользователь выбирает, какие процедуры организация берет в исполнение;
- можно напомнить через час;
- можно делегировать;
- при делегировании сотрудник повышается до роли менеджера по процедуре.

Это похоже на inbox входящих тендеров. Для MVP это может быть экран `New tenders inbox`.

### Проверка участия двумя специалистами

PDF разделяет первичную оценку:

- специалист по работе с государственным сегментом проверяет готовность/возможность участия;
- специалист отдела продаж проверяет соответствие процедуры интересам/возможностям компании;
- при отказе любого специалиста менеджер получает задачу и может продолжить/уточнить.

Возможны дополнительные задачи на получение возможности участия.

### URL-проверка товарных позиций

Есть отдельный механизм:

- генерируется URL;
- на странице доступны поля сделки с редактированием;
- товарные позиции можно добавлять, удалять, редактировать;
- варианты оценки: `Верно`, `Ошибка`, `Грубая ошибка`, `Неверно`, `Заблокировать`;
- после нажатия кнопки URL удаляется.

Это сильное правило безопасности: временные ссылки для внешних/фрилансеров должны быть одноразовыми или ограниченными по времени.

### Топ-3 поставщиков и цены

После каждого ответа поставщика формируется `топ три`.

Действия системы:

- удаление товарных позиций в затратных сделках, не попавших в топ-3;
- удаление затратных сделок без товарных позиций;
- проставление цен в доходной сделке как среднее арифметическое от топ-3;
- если нет 3 первых мест, пересчитывать на 2 или брать единственную предложенную цену.

Это нужно превратить в модуль `QuoteRanking`:

- gold/silver/bronze или top-1/top-2/top-3;
- минимальное, среднее, максимальное предложение;
- источник цены;
- правила fallback, если поставщиков меньше трех.

### Три сценария закупочной себестоимости

PDF задает три расчета минимальной себестоимости закупки:

1. по минимальным предложениям;
2. по средним предложениям;
3. по максимальным предложениям.

Если `НМЦК` больше нуля, система сравнивает `minSum / 0.85` с `НМЦК`.

Если затраты с минимальной наценкой превышают `НМЦК`, система предлагает:

- перевести сделку в `Нерентабельная сделка`;
- продолжить работу в ручном режиме.

Ручной режим:

- все исполнители меняются на менеджера;
- стадия становится `Ручное управление`.

### Статьи затрат из PDF

PDF подтверждает и расширяет список затрат:

- закупка товара/работы/услуги;
- логистика;
- командировка;
- выплата по кредитам/займам/инвестициям;
- бонус сотруднику;
- бонус покупателю;
- аренда офиса/склада/спецтехники/оборудования;
- банковское обслуживание;
- связь;
- налоги;
- отчисления с заработной платы;
- маркетинг и реклама;
- хозяйственные нужды;
- прочее.

Часть затрат вводится человеком, часть рассчитывается автоматически.

### Автоматические бонусы и налоги

PDF содержит формулы:

- бонус менеджеру: `0.5%` от суммы контракта;
- бонус продавцу: `1%` от суммы контракта;
- премиальный фонд: `1.5%` от суммы контракта;
- налог НДС: `(НМЦК - сумма всех затрат) * 0.2`;
- при отсутствии `НМЦК` сумма затрат делится на `0.7`, чтобы получить расчетную базу.

В формулах встречается деление на `0.84` как учет оптимизации вывода/налоговой нагрузки. Для нового продукта такие коэффициенты надо сделать настройками организации, а не зашивать в код.

### Документы-основания и подтверждения затрат

Для затрат есть поля:

- сумма план;
- сумма факт;
- дата план;
- дата факт;
- документ-основание;
- документ-подтверждение;
- согласование;
- исполнено.

Если не заполнена дата плана или не подгружен документ-основание, создается задача ответственному. Если наступила дата плана, требуется прикрепить чек оплаты.

### Обеспечение заявки и контракта

Есть отдельные сценарии:

- обеспечение заявки не требуется;
- обеспечение заявки требуется;
- обеспечение исполнения контракта не требуется;
- обеспечение исполнения контракта требуется.

Ответственный выбирает способ несения затрат. При просрочке часть задач передается менеджеру.

### Подбор поставщиков по ОКПД2/ОКВЭД

Алгоритм из PDF:

1. Итерируем все товарные позиции сделки.
2. Берем поле `ОКПД2`.
3. Извлекаем первые 5 знаков формата `XX.XX`.
4. Сохраняем их в поле видов деятельности.
5. Фильтруем компании типа `Поставщик`.
6. Ищем совпадение с основным и дополнительными кодами ОКВЭД.
7. Добавляем найденные компании в поле поставщиков сделки.
8. Если поставщиков недостаточно, запускаем внешний подбор новых поставщиков по ОКВЭД.
9. Для поставщиков создаются затратные сделки.
10. В затратные сделки копируются только те товарные позиции, которые соответствуют деятельности поставщика.

Для нового ASTS лучше не копировать старую модель `доходная сделка / затратная сделка` буквально, но сохранить идею:

- `Tender` как доходная процедура;
- `SupplierRequest`/`SupplierQuote` как закупочная сторона;
- `QuoteLine` как связь поставщика с конкретной позицией.

### Базовые реквизиты поставщика

PDF описывает проверку:

- юридическое лицо или ИП;
- заполнение полей по таблице реквизитов;
- если ИНН/ОГРН уже есть, обновлять информацию, контакты и реквизиты;
- ИНН/ОГРН могут быть записаны в реквизитах;
- при обновлении писать в комментарии, какие поля обновлены.

Это будущий модуль supplier enrichment.

### Сроки и ресурсная оценка старой доработки

В PDF есть оценки:

- доработка на 1 месяц: 3 junior Python developer, 1 тестер, 1 middle Python developer, 1 DevOps, 1 junior JavaScript backend, бюджет 400;
- доработка на 3 месяца: тот же состав, бюджет 1200.

Единицы бюджета не указаны. Использовать только как историческую оценку масштаба, не как актуальный бюджет.

### Кто подает документы

Отдельный этап:

- вопрос “Кто будет подавать документы на площадку?”;
- варианты: B2G специалист или фрилансер;

## Восстановленный workflow

### 1. Выбор тендерной процедуры

Сотрудник получает задачу по процедуре.

Доступны действия:

- ссылка на процедуру;
- ссылка на документы;
- принять задачу;
- отказаться;
- перевести сделку в нерентабельную;
- проверить работу самостоятельно в ручном режиме.

### 2. Решение об участии

Система предлагает подтвердить участие или отказ.

Причины отказа из интерфейса:

- нерентабельная процедура;
- не уложимся в сроки исполнения;
- очевидное отсутствие рентабельности;
- не интересующая сумма сделки;
- не интересующий заказчик;
- прочее.

Это нужно перенести в новый ASTS как структурированный справочник причин отказа.

### 3. Наполнение товарных позиций

Фрилансер/категорийный менеджер заполняет товарные позиции по процедуре.

Видимые элементы:

- ссылка на страницу заполнения товарных позиций;
- проверка заполненных позиций специалистом отдела закупок;
- подтверждение корректности;
- возможность изменить данные;
- капча/подтверждение перед отправкой.

Для нового MVP это означает:

- нужна таблица `TenderPosition`;
- нужен статус проверки позиции;
- нужен исполнитель и проверяющий;
- нужна история правок.

### 4. Проверка позиций закупщиком

После заполнения фрилансером задача уходит закупщику.

Закупщик:

- проверяет позиции;
- подтверждает корректность;
- может отклонить/попросить исправить;
- фиксирует отказ специалиста отдела закупок.

Это отдельный workflow, не просто поле в карточке тендера.

### 5. Подбор поставщиков

Видны сценарии:

- выбор поставщиков для запроса КП;
- система показывает число выбранных поставщиков;
- если поставщиков меньше 3, уведомление направляется специалисту службы безопасности и менеджеру;
- можно дополнить список получателей собственным списком email;
- можно открыть список компаний, которые получат письма.

Бизнес-правило:

> Если выбрано менее 3 поставщиков, система должна предупреждать и отправлять уведомление ответственным.

### 6. Генерация и отправка запроса КП

В архиве есть PDF `КП (54).pdf`. Это пример запроса коммерческого предложения.

Содержит:

- компанию-запрашивающего: АО “СВЕТ”;
- реквизиты;
- исходящий номер;
- крайний срок предоставления КП;
- текст обращения к поставщику;
- требование указывать цену с учетом НДС;
- разрешение предлагать аналоги при строгом соответствии характеристикам;
- таблицу товаров/работ/услуг;
- условия оплаты;
- регион поставки;
- поставка для нужд конкретного заказчика;
- ответственного за закупку;
- контакты.

Для ASTS это должен быть генератор запроса КП:

- HTML/email-версия;
- PDF-версия;
- публичная ссылка поставщику;
- список позиций;
- дедлайн ответа;
- правила по аналогам;
- условия оплаты/поставки.

### 7. Получение КП от поставщика

Поставщик может:

- заполнить ценовое предложение;
- указать цену;
- указать НДС;
- указать срок действия предложения;
- загрузить счет;
- отправить КП.

Видны отдельные формы:

- “Сохранить и отправить ценовое предложение”;
- “Подгрузить счет или коммерческое предложение”.

Для нового продукта:

- поставщик не должен иметь полноценный личный кабинет на старте;
- достаточно публичной страницы по токену;
- позднее можно добавить регистрацию поставщика.

### 8. Проверка аналогов

В Telegram есть сообщение:

> Поставщик ответил на наше коммерческое предложение и внес цены на запрашиваемые позиции на нашем сайте. Ссылка на страницу для проверки аналогов.

Это важный сценарий:

- поставщик может предложить аналоги;
- закупщик должен проверить соответствие аналогов;
- цены поставщика нельзя тихо менять после отправки;
- решение по аналогу должно быть отдельным статусом.

### 9. Статьи затрат

На скриншотах видны вводы затрат:

- бонус;
- удержание площадкой;
- удержание за участие;
- удержание за победу;
- банковская гарантия;
- обеспечение заявки;
- логистика;
- форма оплаты;
- НДС.

Варианты НДС:

- НДС не облагается;
- +10% НДС;
- в том числе НДС 10%;
- +20% НДС;
- в том числе НДС 20%.

Форма оплаты:

- безналичный расчет;
- наличный расчет.

Эти справочники надо заложить в модель данных и UI.

### 10. Логистика

Есть отдельная ссылка на логистическую задачу:

`/logistic/...`

Видно сообщение:

> В затрате не была заполнена дата плана, либо не был подгружен документ основания, требуется заполнить их.

Значит для затрат нужны:

- сумма;
- дата плана;
- документ-основание;
- ответственный;
- статус заполнения.

### 11. Подача документов

Есть сценарии:

- “Необходимо подать документы в процедуру”;
- “Ссылка для выполнения”;
- добавление предоставляемых документов для ИП/ООО;
- подтверждение перед отправкой.

Для нового MVP это можно не автоматизировать полностью, но нужно учесть как будущий этап воронки.

### 12. Делегирование

В интерфейсе есть:

- выбор, кому делегировать работу;
- варианты “Менеджеры”, “Сотрудники”;
- уведомления при отказе специалиста;
- возможность менеджера принять решение дальше.

Это подтверждает, что ASTS должен быть role/task-driven, а не просто CRM-карточкой.

## Бизнес-правила

1. Нельзя отправлять запрос поставщикам, если позиции не проверены.
2. Если поставщиков меньше 3, уведомить СБ и менеджера.
3. Поставщик может предложить аналог, но закупщик должен подтвердить соответствие.
4. Цена поставщика после отправки должна быть защищена от незаметного изменения.
5. Каждое действие с деньгами требует подтверждения.
6. Затрата должна иметь сумму, тип, дату плана и документ-основание, если это требуется типом затраты.
7. Отказ от участия должен иметь структурированную причину.
8. Нерентабельная сделка - отдельный статус, а не просто комментарий.
9. Telegram-уведомление должно вести на конкретную задачу/страницу.
10. Менеджер может переопределять решения специалистов, но это должно попадать в audit log.
11. Прямой дедлайн подачи заявок контролируется автоматически; если до дедлайна меньше 10 часов, сделка уходит в `Вышел срок`.
12. Просрочка задачи должна запускать политику эскалации, зависящую от роли и типа задачи.
13. После каждого ответа поставщика пересчитывается top-3 предложений.
14. Расчет закупочной себестоимости должен поддерживать минимальный, средний и максимальный сценарии.
15. Если `minSum / 0.85` превышает `НМЦК`, система предлагает `Нерентабельная сделка` или `Ручное управление`.
16. Затраты требуют плановой даты и документа-основания, если это предусмотрено типом затраты.
17. При наступлении плановой даты затраты нужен документ-подтверждение оплаты.
18. Коэффициенты бонусов, налогов и оптимизации должны быть настройками организации.
19. Временные URL для проверки позиций и внешних действий должны иметь срок жизни и статус использования.
20. Подбор поставщиков должен учитывать связь ОКПД2 позиций с ОКВЭД поставщика.

## Что важно для нового AI-слоя

AI может заменить часть ручной работы старой системы:

- извлечь позиции из документации вместо ручного заполнения фрилансером;
- предложить ОКПД2/категорию;
- найти требования и риски;
- сформировать черновик запроса КП;
- нормализовать КП поставщика из PDF/XLSX;
- подсказать, какие позиции требуют проверки аналога;
- обнаружить неполные затраты;
- подготовить карточку решения по участию.

Но старые бизнес-правила остаются полезными: AI помогает, а статусы, подтверждения и audit log защищают процесс.

## Практический вывод

RTB-доска подтверждает, что у проекта уже была сильная операционная логика. Новый ASTS нужно строить не как простой tender parser, а как восстановленную и модернизированную B2G-систему:

> AI + workflow + поставщики + экономика сделки + контроль людей.

Именно это может быть продаваемым отличием от обычных сервисов поиска тендеров.
