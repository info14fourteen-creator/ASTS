# АСТС: проект проекта

## Что строим

АСТС - SaaS-система для управления тендерной продажей.

Главная идея: не конкурировать напрямую с базами поиска тендеров, а закрыть боль после того, как тендер найден:

- быстро разобрать документацию;
- понять, стоит ли участвовать;
- выделить позиции, требования, сроки и риски;
- собрать цены поставщиков;
- сравнить КП;
- посчитать рентабельность;
- поставить задачи людям;
- не пропустить дедлайн.

## Позиционирование

Коротко:

> AI-диспетчер тендерного отдела: от разбора закупки до сбора КП и расчета маржи.

Для клиента:

> Загружаете тендерные документы - система за несколько минут показывает суть закупки, риски, список позиций, задачи команде и предварительную экономику участия.

## На чем пишем

### Frontend

Next.js + TypeScript + React.

Почему:

- удобно делать рабочие кабинеты, таблицы, формы и дашборды;
- сильная экосистема;
- легко нанимать разработчиков;
- можно быстро делать SaaS-интерфейс.

### Backend

Python + FastAPI.

Почему:

- Python сильнее всего для AI, документов, OCR, XLSX/PDF/DOCX, фоновых задач;
- FastAPI дает быстрый typed API и OpenAPI-документацию;
- проще строить AI-пайплайны и парсинг.

### База данных

PostgreSQL + pgvector.

Почему:

- тендеры, поставщики, КП, задачи, пользователи и сметы - это нормальные связанные сущности;
- pgvector позволит искать по смыслу внутри документов и истории без отдельной vector DB на старте.

### Очереди и фоновые задачи

Redis + Celery/RQ.

Нужно для:

- разбора документов;
- AI-анализа;
- рассылок;
- напоминаний;
- эскалаций;
- генерации отчетов.

### Файлы

S3-совместимое хранилище. Локально - MinIO.

Туда кладем:

- исходные документы тендера;
- извлеченный текст;
- счета и файлы поставщиков;
- готовые отчеты.

### AI-слой

Делаем отдельный модуль `ai`, а не раскидываем вызовы модели по коду.

Задачи AI:

- структурное извлечение данных из документов;
- поиск требований и рисков;
- извлечение товарных позиций;
- подсказка ОКПД2/категорий;
- генерация письма поставщику;
- нормализация КП из файлов;
- ответы на вопросы по тендерной документации.

## Как выдаем данные клиенту

### 1. Карточка тендера

Основной экран менеджера:

- название;
- заказчик;
- НМЦК/начальная цена;
- дедлайн;
- площадка;
- статус;
- ответственный;
- AI-рекомендация;
- ближайшее действие.

### 2. Карточка решения

Главный продающий элемент MVP:

- участвовать;
- отказаться;
- нужна ручная проверка;
- аргументы за;
- аргументы против;
- риски;
- что не хватает;
- ссылки на страницы/файлы-источники.

### 3. Реестр требований

Для тендерного специалиста/юриста:

- требования к участнику;
- список документов;
- обеспечение заявки;
- обеспечение контракта;
- сроки;
- ограничения;
- источник каждого пункта.

### 4. Таблица позиций

Для закупщика:

- наименование;
- количество;
- единица;
- характеристики;
- ОКПД2/категория;
- уверенность AI;
- статус проверки.

### 5. Запрос поставщику

Выдаем:

- текст письма;
- публичную ссылку для заполнения КП;
- таблицу позиций;
- срок ответа;
- возможность загрузить счет.

### 6. Сравнение КП

Для закупки и финансов:

- поставщик;
- цена по каждой позиции;
- НДС;
- срок поставки;
- аналоги;
- общий итог;
- лучший/рисковый вариант.

### 7. Расчет рентабельности

Для руководителя:

- доход по тендеру;
- закупочная стоимость;
- логистика;
- удержания площадки;
- обеспечение/финансирование;
- командировки;
- бонусы;
- прочие затраты;
- валовая прибыль;
- маржа.

### 8. Задачи и дедлайны

Для команды:

- кому что делать;
- срок;
- статус;
- напоминания;
- эскалация менеджеру;
- история действий.

### 9. Экспорты

На старте:

- XLSX-отчет по анализу;
- XLSX-сравнение КП;
- PDF-отчет для решения;
- JSON через API.

Позже:

- сделка в Bitrix24/AmoCRM;
- выгрузка для 1C;
- email-дайджест;
- отчет руководителю.

## Первый MVP

Минимальный продаваемый продукт:

1. Импорт тендера из XLSX или создание по ссылке.
2. Загрузка документов.
3. AI-разбор документов.
4. Карточка решения.
5. Таблица позиций.
6. Запросы поставщикам по публичной ссылке.
7. Сравнение КП.
8. Первичный расчет маржи.
9. Telegram-уведомления по задачам.

## Что не делаем в первой версии

- автоподачу заявок на площадки;
- полную интеграцию с 1C;
- биллинг;
- сложную аналитику конкурентов;
- маркетплейс фрилансеров;
- обещания, что AI сам принимает юридически значимое решение.

## Главная проверка рынка

Первый пилот должен доказать:

- сколько времени экономим на разборе одного тендера;
- сколько тендеров можно обработать в неделю;
- насколько быстрее собираются КП;
- сколько нерентабельных тендеров отсекается раньше;
- уменьшается ли количество пропущенных дедлайнов.

## Решение по запуску

Дальше надо делать не презентацию, а рабочий локальный MVP:

- поднять Postgres/Redis/MinIO;
- создать FastAPI;
- создать Next.js;
- сделать импорт Tenderland XLSX;
- сделать карточку тендера;
- сделать загрузку файлов;
- подключить первый AI-разбор.
