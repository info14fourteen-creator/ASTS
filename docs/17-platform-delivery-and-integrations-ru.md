# План платформы, доступа и интеграций

Дата обсуждения: 2026-06-11 01:50 +05:00.

Дата фиксации: 2026-06-12.

## Что решили

ASTS должен быть не только web-кабинетом, а полноценной платформой с несколькими каналами доступа:

- публичный сайт на основном домене;
- полный web-доступ через `app.site.ru`;
- адаптивный web для desktop, notebook, tablet и mobile browser;
- мобильные приложения в App Store и Google Play;
- Telegram Bot;
- Telegram Mini App;
- интеграции с CRM и учетными системами.

## Принципы из переговоров

### Источники данных

Данные по закупкам строим на первоисточниках:

- ЕИС / zakupki.gov.ru;
- официальные API и выгрузки электронных торговых площадок;
- ФНС и официальные открытые данные;
- официальные реестры.

Агрегаторы вроде TenderLand, Контур, Seldon и похожих сервисов не являются источником истины. Их можно использовать только для сравнения, миграции старых файлов или проверки полноты.

### Хранилище и фильтрация

Большие таблицы и документы не фильтруем “на лету” из внешних источников. Сначала забираем raw-данные, сохраняем у себя, нормализуем в PostgreSQL и только потом даем быстрый поиск/фильтры.

Слои:

- raw layer: оригинальные HTML/XML/JSON/PDF/DOCX/XLSX/ZIP;
- normalized layer: процедуры, заказчики, позиции, документы, контракты, поставщики;
- search layer: SQL-индексы, полнотекстовый поиск, embeddings;
- decision layer: AI-карточки, риски, рекомендации, workflow.

### Onboarding клиента

При подключении компании система должна собрать начальные условия, чтобы рекомендации были не общими, а персональными.

Минимальный профиль:

- ИНН/ОГРН компании;
- регионы работы;
- категории товаров/работ/услуг;
- ОКПД2/ОКВЭД интересов;
- исключения и стоп-темы;
- минимальная/максимальная НМЦК;
- желаемая маржинальность;
- правила НДС;
- доступные поставщики;
- текущие CRM/учетные системы;
- роли сотрудников;
- кто принимает решения;
- кто отвечает за закупку/логистику/документы;
- действующие шаблоны КП и писем.

## Каналы доступа

### 1. Публичный сайт

Назначение:

- объяснить продукт;
- показать ценность для тендерного отдела;
- собрать лиды;
- дать вход в приложение;
- дать материалы для переговоров.

Не должен содержать закрытую логику, секреты, реальные клиентские данные и внутренние формулы.

### 2. Web app: `app.site.ru`

Основной рабочий кабинет.

Требования:

- desktop-first для тендерных отделов;
- корректная работа на notebook;
- адаптивность для tablet;
- usable mobile browser для срочных approvals;
- единый URL входа;
- multi-tenant организация;
- роли и права;
- audit log;
- загрузка и просмотр документов;
- две воронки сделки: до победы и после победы;
- карточка решения;
- задачи и дедлайны;
- CRM sync status.

Технологический базис:

- Next.js;
- TypeScript;
- React;
- responsive app shell;
- API через FastAPI.

### 3. Mobile web

Mobile browser не должен быть отдельным урезанным продуктом. Он нужен для:

- просмотра карточки решения;
- подтверждения действий;
- дедлайнов;
- комментариев;
- быстрых согласований;
- просмотра статуса поставщиков/КП.

Полная табличная работа остается удобнее в desktop/tablet.

### 4. iOS и Android приложения

Назначение:

- push-уведомления;
- быстрые решения;
- согласования затрат;
- контроль дедлайнов;
- комментарии;
- просмотр карточки процедуры;
- менеджерский контроль без открытия desktop.

Публикация:

- App Store;
- Google Play.

Рекомендуемый путь:

- сначала responsive web + Telegram Mini App;
- затем Expo/React Native приложение на общей TypeScript-базе;
- native-only функции добавлять только при необходимости.

### 5. Telegram Bot

Назначение:

- push-уведомления;
- быстрые кнопки;
- deep links;
- напоминания;
- статусы задач.

### 6. Telegram Mini App

Назначение:

- “операционный пульт” внутри Telegram;
- мои задачи;
- карточка процедуры;
- подтверждение поставщиков;
- согласование затрат;
- просмотр КП;
- загрузка файлов, если Telegram UI это позволяет удобно;
- быстрый переход в web app.

## CRM и учетные интеграции

### Цель

Не заставлять компании ломать текущий тендерный отдел и CRM. ASTS должен подключаться постепенно:

1. сначала показываем добавленную ценность рядом с существующим процессом;
2. потом синхронизируем сделки, компании, контакты и задачи;
3. затем добавляем виджеты/приложения внутри CRM;
4. только после доверия переносим больше workflow в ASTS.

### CRM-кандидаты

Первая очередь:

- Bitrix24;
- amoCRM;
- 1C.

Вторая очередь:

- Мегаплан;
- RetailCRM;
- Planfix;
- СберCRM;
- C2CRM;
- РосБизнесСофт CRM;
- BPMSoft CRM.

Перед реализацией каждого коннектора нужно отдельно проверить официальную документацию API, лимиты, OAuth/токены, webhooks, marketplace-правила и условия использования.

### Единая архитектура CRM hub

Не пишем бизнес-логику отдельно под каждую CRM.

Нужен общий слой:

```text
CrmProvider
  -> authenticate()
  -> pull_companies()
  -> pull_contacts()
  -> pull_deals()
  -> push_deal()
  -> push_task()
  -> push_comment()
  -> sync_status()
  -> map_fields()
```

Внутри ASTS:

- `crm_connections`;
- `crm_field_mappings`;
- `crm_sync_jobs`;
- `crm_sync_events`;
- `external_entity_links`;
- `integration_secrets`;
- `webhook_events`.

### Уровни интеграции

#### Level 1: Export/link mode

Самый безопасный старт:

- ASTS создает карточку/ссылку;
- CRM получает ссылку на процедуру;
- можно создать задачу ответственному;
- данные не перетираются в CRM.

#### Level 2: One-way sync

ASTS отправляет:

- сделку/лид;
- компанию;
- контакт;
- задачу;
- комментарий;
- статус;
- ссылку на документы.

#### Level 3: Two-way sync

ASTS и CRM обмениваются:

- статусами;
- ответственными;
- задачами;
- комментариями;
- файлами;
- изменениями компаний/контактов.

Нужны conflict rules и audit log.

#### Level 4: CRM app/widget

Внутри CRM появляется виджет:

- AI-карточка процедуры;
- статус воронки ASTS;
- кнопка открыть процедуру;
- top-3 поставщиков;
- рентабельность;
- дедлайны.

## 1C и учетные системы

На старте не делаем прямую банковскую интеграцию.

Полезная интеграция с учетом:

- контрагенты;
- счета;
- оплаты план/факт;
- закрывающие документы;
- номенклатура;
- остатки/склады, если клиенту нужно;
- сверка фактических затрат.

Для первых клиентов можно начать с файлового обмена или полуавтоматического импорта, а полноценный обмен с 1C вынести в отдельный этап.

## Как продавать компаниям с действующим тендерным отделом

Не продавать как “замену вашего отдела”.

Позиционирование:

- AI-помощник тендерного отдела;
- система контроля дедлайнов;
- ускоритель разбора документации;
- единая база процедур, документов и КП;
- прозрачная экономика сделки;
- меньше ручной рутины;
- меньше потерь из-за сроков и незамеченных требований.

Тактика внедрения:

1. Shadow mode: ASTS анализирует процедуры рядом с текущим процессом.
2. Pilot: одна категория или один менеджер.
3. Supplier quote workflow: показать экономию времени на КП.
4. Profitability card: показать деньги и риски.
5. CRM sync: подключить к текущему процессу без ломки.
6. Расширение на весь тендерный отдел.
