# Zakupki.gov.ru / ЕИС: чеклист данных для импорта тендера

Цель документа - быстро проверить, могут ли разработчики получить из
zakupki.gov.ru / ЕИС данные, необходимые для импорта тендера, скачивания
документов и дальнейшего AI-разбора.

## Что надо проверить

Разработчикам нужно ответить на три вопроса:

1. Можно ли по номеру закупки или ссылке получить карточку закупки?
2. Можно ли получить список документов закупки и скачать сами файлы?
3. Можно ли получить или извлечь позиции, требования, сроки и условия,
   достаточные для создания карточки тендера и запуска AI-разбора?

Если все три пункта возможны, интеграция подходит для MVP.

## Возможные источники данных

Проверить доступные способы:

- официальный сервис отдачи информации/документов ЕИС;
- открытые данные ЕИС;
- XML/print-form карточки закупки, если доступна;
- публичные страницы zakupki.gov.ru как fallback;
- скачивание документации по ссылкам из карточки закупки.

Важно: у ЕИС менялись способы доступа, поэтому код надо проектировать через
заменяемый адаптер, а не привязывать всю систему к одному endpoint.

## Минимум данных для импорта тендера

Для создания базовой карточки тендера нужны поля:

- номер закупки / извещения;
- ссылка на закупку в ЕИС;
- закон/тип закупки: 44-ФЗ, 223-ФЗ или другой источник;
- способ закупки / процедура;
- статус закупки;
- название закупки;
- заказчик;
- ИНН заказчика;
- КПП/ОГРН заказчика, если доступны;
- НМЦК / начальная цена;
- валюта;
- дата публикации;
- срок подачи заявок;
- площадка;
- регион или место поставки;
- дата последнего обновления в источнике.

Эти данные должны позволить создать запись `Tender`.

## Документы закупки

Нужно получить список документов:

- идентификатор документа;
- название документа;
- тип документа;
- имя файла;
- ссылка на скачивание;
- MIME type или расширение;
- размер файла, если доступен;
- дата публикации;
- дата обновления или версия, если доступна.

Нужно проверить скачивание файлов:

- PDF;
- DOC/DOCX;
- XLS/XLSX;
- ZIP/RAR/7z, если встречаются;
- XML;
- изображения/сканы, если встречаются.

Оригинальные файлы должны сохраняться без изменений в S3/MinIO. В базе должны
храниться только метаданные и ссылка на объект в хранилище.

## Позиции закупки

Проверить, доступны ли структурированные позиции из ЕИС:

- наименование позиции;
- описание;
- количество;
- единица измерения;
- ОКПД2;
- КТРУ, если есть;
- цена за единицу;
- сумма позиции;
- номер лота;
- характеристики;
- место/срок поставки по позиции.

Если структурированных позиций нет или они неполные, нужно проверить, можно ли
извлечь позиции из документов закупки через AI/OCR/parser.

## Требования и условия

Нужно получить или извлечь:

- требования к участнику;
- список обязательных документов для заявки;
- лицензии, СРО, сертификаты;
- требования к опыту;
- обеспечение заявки;
- обеспечение исполнения контракта;
- гарантийные обязательства;
- условия оплаты;
- условия поставки;
- сроки поставки/исполнения;
- штрафы и пени;
- ограничения, преимущества, национальный режим;
- основания для отказа или риски.

Эти данные нужны для `TenderRequirement`, `TenderRisk` и карточки решения.

## Лоты

Если закупка содержит лоты, нужно проверить:

- номер лота;
- название лота;
- НМЦК лота;
- статус лота;
- позиции лота;
- место поставки;
- сроки.

Для MVP можно начать с одного лота, но формат данных должен не ломаться на
много-лотовых закупках.

## История изменений

Проверить, можно ли понять, что закупка обновилась:

- версия извещения;
- дата изменения;
- причина изменения;
- измененные документы;
- новый срок подачи заявок;
- новая цена, если менялась.

Минимум для MVP:

- `source_updated_at`;
- `source_version`;
- флаг `needs_reanalysis`, если документы или ключевые поля изменились.

## Данные для AI-разбора

AI-модулю нужны:

- оригинальные документы;
- извлеченный текст документов;
- номера страниц/листов;
- структурированные поля ЕИС;
- карточка закупки;
- позиции из ЕИС, если есть;
- НМЦК;
- дедлайн;
- заказчик;
- место поставки.

AI должен вернуть:

- факты по закупке;
- требования;
- риски;
- позиции;
- недостающие данные;
- рекомендацию: участвовать, отказаться или ручная проверка;
- ссылки на источники: файл, страница, лист, фрагмент текста;
- confidence score.

## Что разработчики должны предоставить после проверки

Нужен короткий технический ответ:

1. Какие способы доступа проверены.
2. Нужна ли регистрация/токен/ЕСИА.
3. Какие endpoint или URL реально работают.
4. Какие данные доступны структурированно.
5. Какие данные доступны только из документов.
6. Какие документы можно скачать автоматически.
7. Какие ограничения обнаружены: капча, авторизация, лимиты, нестабильные ссылки.
8. Какие форматы приходят: XML, JSON, HTML, ZIP, PDF, DOCX, XLSX.
9. Можно ли сделать MVP-импорт по номеру закупки.
10. Что остается риском.

## Минимальный успешный результат spike

Spike считается успешным, если разработчик может взять один реальный номер
закупки и получить:

- карточку закупки;
- заказчика;
- НМЦК;
- дедлайн подачи;
- ссылку на ЕИС;
- список документов;
- минимум один скачанный документ;
- сырой XML/HTML/JSON или другой raw-source;
- извлеченный текст из скачанного документа;
- черновой список позиций или объяснение, почему позиции доступны только через AI.

## Вывод

Для MVP достаточно доказать, что по номеру закупки можно автоматически получить
карточку, документы и хотя бы часть позиций/требований. Все, что не удается
получить структурированно, можно передавать в AI-разбор через документы, но
оригинальные файлы и raw-ответы источника обязательно надо сохранять.
