# Политика первоисточников данных

Дата: 2026-06-11.

## Принцип

ASTS не должен зависеть от агрегаторов как от основного источника данных.

Источник истины для продукта:

- официальные государственные источники;
- официальные API/выгрузки электронных торговых площадок;
- официальные реестры и открытые данные;
- документы, скачанные из первичного источника и сохраненные у нас с контрольной суммой.

Агрегаторы можно использовать только как:

- исторический импорт старых выгрузок;
- временный мост для сравнения полноты данных;
- ручной файл от клиента;
- источник гипотез, но не источник истины.

## Разрешенные первоисточники для первой архитектуры

### ЕИС / zakupki.gov.ru

Используем как основной источник закупок и документов:

- поиск закупок 44-ФЗ и 223-ФЗ;
- карточки извещений;
- документация закупки;
- реестр контрактов 44-ФЗ;
- реестр договоров 223-ФЗ;
- реестр организаций;
- КТРУ;
- реестр недобросовестных поставщиков;
- перечень отобранных операторов электронных площадок;
- открытые данные ЕИС.

В коде это отдельный модуль `data_sources/eis`.

### ФНС / nalog.gov.ru

Используем для юридической проверки организаций и поставщиков:

- ЕГРЮЛ/ЕГРИП через официальный режим интеграции;
- официальные открытые данные ФНС;
- сервисы ФНС по контрагентам, если условия использования позволяют автоматическую интеграцию;
- ФИАС/адресные данные, если нужно нормализовать адреса.

В коде это отдельный модуль `data_sources/fns`.

### Официальные электронные торговые площадки

Подключаем площадки как независимые адаптеры, если у них есть официальный API, выгрузка, личный кабинет или разрешенный способ интеграции:

- Сбербанк-АСТ;
- РТС-тендер;
- Росэлторг;
- ТЭК-Торг;
- ЭТП ГПБ;
- АГЗ РТ;
- АСТ ГОЗ;
- другие площадки из официального перечня ЕИС.

В коде это отдельные адаптеры `data_sources/etp/*`.

## Запрещено как основа продукта

Нельзя строить core-логику на данных агрегатора, если те же данные можно получить из первоисточника:

- Tenderland;
- Контур.Закупки;
- Seldon;
- Tenderplan;
- Tendee;
- любые парсеры третьих лиц;
- чужие базы поставщиков без подтверждения источника.

Исключение: разовый импорт старых клиентских файлов или платная проверка полноты данных. В таком случае запись получает флаг `source_kind = aggregator_import`, а не `official_source`.

## Требования к хранению

Для каждой записи храним:

- `source_system` - ЕИС, ФНС, конкретная ЭТП;
- `source_url` или идентификатор API;
- `external_id` - номер извещения, ИНН, ОГРН, номер контракта;
- `fetched_at`;
- `raw_payload_path`;
- `raw_payload_hash`;
- `normalization_version`;
- `source_terms_snapshot` - ссылка/заметка об условиях доступа, если требуется;
- `confidence` - уровень уверенности нормализации, а не уверенность в первичном факте.

Raw-данные не перезаписываем. Если источник изменился, сохраняем новую версию.

## Архитектурное правило

Каждый источник подключается через интерфейс:

```text
OfficialSourceProvider
  -> fetch_index()
  -> fetch_entity(external_id)
  -> fetch_documents(external_id)
  -> normalize(raw_payload)
  -> persist_raw(raw_payload)
```

В бизнес-логике ASTS не должно быть кода, завязанного на HTML ЕИС, формат конкретной площадки или файл Excel. Все это живет в адаптерах источников.

## Приоритет подключения

1. ЕИС: закупки, карточки, документы, реестры контрактов.
2. ФНС: ЕГРЮЛ/ЕГРИП и проверка юридического лица.
3. Официальный перечень ЭТП из ЕИС.
4. 2-3 площадки с наибольшей практической ценностью для первых клиентов.
5. Импорт старых Excel/PDF как миграционный инструмент, не как основной источник.
